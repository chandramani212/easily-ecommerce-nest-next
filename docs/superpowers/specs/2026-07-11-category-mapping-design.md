## Original design (approved)

**Status:** Approved (design)

## Problem

The ASI Central import has populated **110,890 products**, but they are
**uncategorized on the storefront**:

- `Category` (curated storefront tree): **0 rows**
- `SourceCategory` (captured from ASI during import): **1,118 rows**, **0 mapped**
- Products with any category: **0**

The mapping backbone already exists in the schema:
`SourceCategory.categoryId` → `Category` (many source cats → one curated cat),
and the import's `resolveCategoryIds` attaches curated categories to a product
**from that mapping** at import time.

**The gotcha:** curated categories are attached to a product only *during
import*, and there is **no stored link from an existing product to its source
categories** (`SourceProductLink` has no category field; `Product` has no
source-category relation). So mapping source→curated today categorizes **future**
imports but does **not** retroactively categorize the 110k products already in
the DB. A separate backfill is required.

## Goal

Give existing products a browsable storefront taxonomy:

1. Build a curated **3-level** category tree, informed by three competitor sites
   (everythingpromo.com, brandedpromo.com, printed4you.co.uk).
2. Map the 1,118 ASI source categories into the tree's **leaf** nodes.
3. Create **only** curated nodes that receive a mapping (hybrid rule), plus a
   small explicit `alwaysCreate` set (currently just `Bestsellers`); skip source
   cats that fit nowhere.
4. Backfill the 110,890 existing products so they appear under their leaf
   categories.

## Key storefront constraint (verified)

From `apps/web/app/[slug]/page.tsx`:

- A category **with children** renders **subcategory tiles**, not products.
- A **leaf** category (no children) renders **products**, filtered by **exact
  `categoryId`** via `/products?categoryId=…` — **no descendant expansion**
  (`apps/api/src/products/products.service.ts:97` → `where.categories = { some: { id } }`).

**Consequence:** a product only needs to be connected to the **leaf** category it
belongs to. Browsing L1→L2→L3 is tile navigation until the leaf. Therefore
**every source category must map to a leaf node**, and the backfill connects
products to leaves only (no ancestor connections).

## Design decisions (approved)

| Decision | Choice |
|---|---|
| Taxonomy source | **Hybrid** — competitor sites shape the top-level structure/naming; a curated node is created only if ≥1 source cat maps to it |
| Competitor categories | **I scrape live** (WebFetch), show merged list for approval |
| Tree depth | **3 levels max** (branches may stop at L2 where ASI has no sub-split); mapped targets are always leaves |
| Backfill | **Search-driven** — per mapped source cat, query ASI search filtered by category, match to existing products |
| Review + apply | **Reviewable checked-in file + idempotent script**; admin mapping UI deferred |

## Components

### A. Competitor taxonomy harvest *(input step, not code)*

WebFetch the nav/sitemap of the three competitor sites, extract their L1/L2
menus, merge into a proposed skeleton that shapes the curated tree's top levels
and naming. Output is shown to the user; it feeds the `curatedTree` draft.

### B. Mapping file — `apps/api/prisma/seed-data/category-map.ts`

The reviewable artifact. Two typed, checked-in structures:

- **`curatedTree`** — nested nodes:
  ```ts
  interface CuratedNode {
    slug: string;      // unique, storefront slug
    name: string;
    sortOrder?: number;
    children?: CuratedNode[];
  }
  ```
  "3-level" is the *maximum* depth. Mapped targets are always **leaves**.

- **`sourceMap`** — `Record<asiExternalId, curatedLeafSlug>`, one entry per
  source cat, with the source name in a trailing comment for review readability:
  ```ts
  export const sourceMap: Record<string, string> = {
    'T00250003': 't-shirts',        // "T-shirts"
    'T00252003': 't-shirts-mens',   // "T-shirts › Mens"
    // 'bestseller...': omitted → skipped
  };
  ```
  Keyed by ASI `externalId` (stable). **Omission = skip.** I generate the full
  1,118-entry draft (each mapped-or-omitted); the user reviews/edits.

### C. Apply script — `apps/api/scripts/apply-category-map.ts`

Idempotent, run inside a Nest application context (reuses `PrismaService`,
`AsiCentralFetcher`, `SecretsCipher`). Four phases:

1. **Validate** — every `sourceMap` target exists in `curatedTree` **and is a
   leaf** (has no children); warn on source `externalId`s absent from the DB.
   Abort on validation error (never create products under a non-leaf).

2. **Create curated categories (hybrid prune + alwaysCreate)** — compute the set
   of used leaf slugs + their ancestors, **unioned with the `alwaysCreate` set**
   (currently `['bestsellers']`, a top-level empty leaf populated manually
   later); upsert only those `Category` rows by unique `slug` (set `name`,
   `parentId`, `sortOrder`, `active: true`). All other tree nodes are never
   created → no accidental empty categories on the storefront.

3. **Set mappings** — write `SourceCategory.categoryId` for each mapped source
   cat (resolved `sourceId` + `externalId` → the leaf slug's `Category.id`).
   Unmapped source cats keep `categoryId = null`.

4. **Backfill products (search-driven)** — for each mapped source cat:
   - list its ASI product ids via the fetcher's category-scoped id collection
     (`category:<token>` selection + price bisection for >1000-product cats,
     reusing existing `collectIds`/`partitionByPrice` primitives);
   - look up `SourceProductLink(sourceId, externalId = asiId)` → `productId`;
   - `product.categories.connect` the leaf `Category`.
   - Batched; idempotent (`connect` is a no-op when already linked).

### D. Future imports — no change required

Existing `resolveCategoryIds` already reads `SourceCategory.categoryId`, so once
the mapping is applied, **new imports auto-categorize**. This feature is a
one-time backfill plus a mapping that keeps paying off.

## Testing (TDD)

- **Prune logic** — only used leaves + their ancestors are created; unused nodes
  are not.
- **Leaf-target validation** — a `sourceMap` value pointing at a parent node is
  rejected.
- **Mapping** — `SourceCategory.categoryId` set correctly; unmapped stay null.
- **Backfill matching** — product connected via `asiId → SourceProductLink →
  product` (mocked fetcher returning ids).
- **Idempotency** — running the script twice yields no duplicate categories and
  no duplicate connections.

## Known risk to verify early (spike, not a blocker)

The backfill's category search needs a token ASI's search `q=category:<…>`
accepts. We stored `SourceCategory.externalId`, which may **not** equal the facet
`ContextPath` that search expects.

**Mitigation (in phase 4):** first pull `dl=category_all` facets (one call) to
build an `externalId`/`name` → `ContextPath` map, then filter by `ContextPath`.
Confirm this resolution works against a single mapped category before running the
full backfill.

## Scope / YAGNI

- **No** admin mapping UI (deferred; possible follow-up phase).
- **No** refresh of existing product data — this touches categories only.
- `Bestsellers` is created as an empty top-level leaf (via `alwaysCreate`) and
  populated **manually** later — the ASI feed has no bestseller/popularity signal
  to auto-fill it. A "featured products" admin picker is a possible follow-up.
